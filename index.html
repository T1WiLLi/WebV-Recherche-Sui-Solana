La trilemme des blockchains

  • • Sécurité, décentralisation, évolutivité.
  • • Souvent deux axes optimisés au détriment du troisième.
  • • Les choix d’architecture matérialisent ce compromis.

Diagramme radar des compromis

Axes de la trilemme

Sécurité

Finalité forte, résistance aux attaques.

Décentralisation

Beaucoup de validateurs, barrières d’entrée faibles.

Évolutivité

Débit élevé, faible latence, coûts contenus.

Ethereum dans la trilemme

  • • Priorise sécurité + décentralisation au niveau L1.
  • • Scalabilité déléguée aux L2 (rollups, validiums).
  • • Matériel de validation relativement accessible.

Repères rapides

Temps de bloc ≈ 12s (slots 12s, epoch ~6,4 min).

Staking min 32 ETH (solo), >1M validateurs (2025).

Couche L2 : >2000 TPS agrégés (estimation rollups).

Solana dans la trilemme

  • • Priorité : haut débit L1.
  • • Matériel de validation plus puissant requis.
  • • Décentralisation suffisante visée malgré blocs denses.

Repères rapides

Temps de bloc cible ~400-600 ms.

TPS observés (moyenne réseau) : 2k–5k, pics >>100k.

Config valideur typique : CPU 12+ cores, 128 Go RAM, NVMe.

Sui dans la trilemme

  • • Modèle objet pour paralléliser les transactions.
  • • BFT et modularité consensus/exécution/stockage.
  • • Cherche un équilibre débit + sécurité.

Repères rapides

Finalité rapide via Mysticeti (<200-400 ms cible).

TPS labo annoncés > 100k (parallélisme objet).

Gas bas pour objets simples (owned) sans consensus complet.

Ethereum – origines

  • • Lancé en 2015 par Vitalik Buterin et des cofondateurs.
  • • Objectif : au-delà du transfert de valeur (smart contracts).
  • • Vision : une « machine virtuelle mondiale » programmable.

Solana – origines

  • • Anatoly Yakovenko + Solana Labs.
  • • Whitepaper centré sur Proof of History (PoH).
  • • Ambition : L1 haute performance pour usages massifs.

Sui – origines

  • • Mysten Labs, ex-équipe Diem (Meta).
  • • Mainnet 2023.
  • • Modèle objet + Move pour simplifier la gestion d’actifs.

Ethereum aujourd’hui

Consensus

PoS + LMD-GHOST

Finalité via Gasper, sécurité forte.

Rôle L1

Couche de règlement

Ancre de sécurité pour les L2.

Scalabilité

Déportée L2

Rollups optimistes / zk en première ligne.

Rôle des L2

Rollups optimistes

Fraude proofs, délais de challenge, coûts modérés.

zk-rollups

Valid proofs, finalité rapide, coût data majoritaire.

Impact dev

Choix du L2 = compromis coûts/latence/compat.

Données

Compressées sur L1, exécution déportée.

PoH + Tower BFT

PoH

Horloge cryptographique pour ordonner sans synchro continue.

Tower BFT

Votes + lockouts qui pénalisent le changement de fork.

Sealevel : exécution parallèle

  • • Transactions déclarent comptes lus/écrits.
  • • Transactions indépendantes en parallèle.
  • • Très efficace pour DeFi/NFT à gros volume.

Mental model

Déclarer les comptes ≈ locking précis → moins de contention.

SPL / Token-2022

  • • SPL = standard de tokens natif (équivalent ERC).
  • • Token-2022 ajoute intérêts, frais, hooks.
  • • Primitives DeFi et NFT directement dessus.

Sui en bref

Modèle objet

Tout est objet possédé ou partagé, donc moins de gros contrats monolithiques.

Parallélisme

Les transactions qui touchent des objets différents passent en parallèle.

Move

Langage pensé pour la sécurité des ressources et la composition d’objets.

Modèle objet – principes

  • • Tout est objet : tokens, coffres, marketplaces.
  • • Propriété explicite (owned, shared, immutable).
  • • Transactions déclarent les objets modifiés.

Owned vs Shared objects

  • • Owned : un seul propriétaire (ex : NFT).
  • • Shared : utilisé par plusieurs (ex : DEX).
  • • Objets disjoints = exécution parallèle sans blocage.
  • • Certaines opérations simples sans consensus complet.

Parallélisme grâce au modèle objet

  • • Transactions sur objets distincts ne se bloquent pas.
  • • Débit dépend de la contention sur un même objet.
  • • Préférence pour des objets granulaires vs contrats monolithiques.

Mysticeti (successeur Narwhal/Bullshark)

  • • Remplace Narwhal/Bullshark, reste basé DAG.
  • • Moins de tours de messages → latence réduite.
  • • Finalité plus rapide, CPU mieux utilisé.

Langage Move sur Sui

Ressources

Pas de double dépense accidentelle, ownership strict.

Modules & structs

Conçus pour le modèle objet (owned/shared).

Capacités

Contrôler qui peut transférer, brûler ou muter un objet.

Sécurité VM

Vérifications de type et de ressources à la compile et à l’exécution.

Exemple Move (objet coin possédé)

// Module simple sur Sui
module my_coin::mint {
    use sui::coin;
    use sui::tx_context::{Self, TxContext};

    public entry fun mint(
        recipient: address,
        ctx: &mut TxContext
    ) {
        let new_coin = coin::mint(1000, ctx); // 1000 unités
        coin::transfer(new_coin, recipient, ctx);
    }
}

Standards de contrats (Ethereum)

  • • Solidity / Vyper.
  • • ERC-20, ERC-721, ERC-1155…
  • • Tooling massif : OpenZeppelin, Foundry, Hardhat.

Programmes on-chain (Solana)

  • • Rust via SDK Solana.
  • • Anchor simplifie macros/IDL.
  • • Comptes remplacent le storage global EVM.

Move et standards d’objets (Sui)

  • • Modules Move déployés on-chain.
  • • Standards basés sur objets (coins, NFT…), pas ERC.
  • • Objets composables : un objet peut en posséder d’autres.

Portage Ethereum ↔ autres

EVM → Solana

Réécriture en Rust/Anchor, penser comptes explicitement et exécution parallèle.

EVM → Sui

Repenser les assets en objets owned/shared et coder en Move.

Rester en EVM

Choisir une sidechain ou L2 EVM pour limiter la réécriture.

Compatibilité Solana

  • • Architecture très différente de l’EVM.
  • • Repenser stockage et accès aux données.
  • • Couches EVM expérimentales mais non natives.

Compatibilité Sui

  • • Modèle objet + Move = changement de paradigme.
  • • Remodeler les assets en objets owned/shared.
  • • Plus de travail initial, mais modèles d’actifs plus riches.

Incidents Ethereum

  • • The DAO, bugs de contrats, forks correctifs.
  • • Surface d’attaque énorme malgré audits.
  • • Importance tooling formel et bug bounties.

Interruptions réseau (Solana)

  • • Plusieurs arrêts depuis le lancement.
  • • Bugs runtime, congestion, ordonnancement.
  • • Culture performance → cycle de risques rapide.

Décentralisation & matériel

  • • Ethereum : entrée validateur plus accessible.
  • • Solana / Sui : besoins matériels plus élevés.
  • • Trade-off performance vs barrière d’entrée.

Comment lire les TPS ?

À garder en tête

TPS théorique ≠ usage réel. Les transactions simples gonflent le chiffre.

Finalité (latence) est tout aussi importante que le débit.

Lecture rapide

Demander la méthodo (observé ou labo ?), le type de tx mesurée et le coût utilisateur.

Vérifier si le réseau reste fluide quand plusieurs applis tournent en même temps.

Performance Ethereum

  • • L1 limitée mais sécurisée.
  • • Charge externalisée sur de multiples L2.
  • • Pas d’ambition de tout traiter sur L1.

Solana & Sui – performance

  • • Solana : PoH + Tower + Sealevel → finalité rapide L1.
  • • Sui : parallélisme objet + Mysticeti réduit la latence.
  • • Solana exige réseau/matériel costaud ; Sui optimise contention.

Expérience dev (Ethereum)

  • • Tooling très mature (Hardhat, Foundry, Truffle).
  • • Templates, libs, docs massives.
  • • Fragmentation possible mais forte communauté.

Expérience dev (Solana)

  • • Rust + Anchor : robuste, mais courbe réelle.
  • • Outils : Solana CLI, explorers, indexation.
  • • Debug plus complexe (parallélisme + comptes).

Expérience dev (Sui)

  • • Sui CLI, Explorer, SDK TypeScript dApps.
  • • Move impose discipline stricte sur les actifs.
  • • Docs en progression, écosystème jeune mais dynamique.

Choisir sa L1 comme dev

  • • Ethereum : compatibilité maximale + L2 variées.
  • • Solana : performance brute pour temps réel massif.
  • • Sui : modèle objet idéal pour actifs complexes parallélisés.

Synthèse

Ethereum

Sécurité et compatibilité max, échelle via L2.

Solana

L1 rapide pour gros volumes, mais matériel plus costaud.

Sui

Modèle objet lisible et parallèle, idéal pour assets composables.

Choisir la L1 = choisir la contrainte principale : compat, perf brute ou modèle objet.

Repères rapides

Ethereum

Priorise sécurité et compatibilité, échelle via L2.

Solana

L1 rapide pour flux soutenus, demande un matériel plus solide.

Sui

Modèle objet et finalité rapide, bonne lisibilité des assets.

Privilégier la fluidité d’explication : chaque réseau a un pari simple, le reste en découle.
